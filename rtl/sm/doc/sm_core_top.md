# sm_core_top 模块说明

本文档说明 `../sm_core_top.v` 在 GPU 实验工程中的定位、内部架构、数据与控制流，以及顶层端口的语义。实现以当前 RTL 为准，并单独列出与完整 SoC 集成时尚未接线的部分。

---

## 1. 模块定位与作用

`sm_core_top` 是 **Streaming Multiprocessor（流式多处理器，SM）** 的顶层封装，在结构上对应 GPU 中负责执行线程束（warp）指令、维护每 warp 寄存器状态、并通过 LSU 访问片上级缓存（本设计中为 L1 类接口）的计算核心。

主要职责包括：

1. **任务级握手**：从上层 GPC（或等效调度器）接收入口 PC、起始 warp 编号等，报告本 SM 是否可接受新任务，以及全部 warp 是否已结束。
2. **指令流水线集成**：将 **前端**（取指/调度/记分板/译码与读寄存器）与 **后端**（译码后路由到各执行单元、写回）通过 **多端口寄存器堆** 连接起来。
3. **异构执行单元**：实例化 ALU、SFU、LSU、LDS 等单元，完成算术与特殊函数、全局访存、共享存储等操作。
4. **观测接口**：输出已发射指令计数与各 warp 活跃状态，便于调试与性能分析。

当前文件注释为 “SM Top Level, integrates all sub-modules”，与上述职责一致。

---

## 2. 参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `NUM_WARPS` | 32 | 支持的 warp 数量上限（与 `warp_status` 位宽、寄存器堆按 warp 分区等一致）。 |
| `NUM_LANES` | 32 | 每个 warp 的 SIMD 通道数（典型为 32 线程/warp）。 |
| `WARP_ID_W` | 5 | `log2(NUM_WARPS)`，用于编码 warp 编号。 |

数据通路上，单 warp 向量操作数常打包为 `32 lanes × 32 bit = 1024 bit`，与 `register_file` 及后端接口一致。

## 3. 架构概览

### 3.1 子模块关系

```
                    ┌─────────────────┐
  GPC 任务 ────────►│   sm_frontend   │◄──── 写回/分支 (wb_*)
  (entry_pc 等)     │ 调度/记分板/译码 │────► issue_* (至后端)
                    └────────┬────────┘
                             │ rf 读 (4 口)
                    ┌────────▼────────┐
                    │ register_file │
                    └────────▲────────┘
                             │ rf 写 (2 口)
                    ┌────────┴────────┐
                    │   sm_backend    │──► ALU / SFU / LSU / LDS 控制
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
   alu_pipeline           sfu                 lsu ──► L1 请求/响应
         │                   │                   │
         └───────────────────┴───────────────────┘
                             │
                         lds_unit（片内共享存储路径）
```

### 3.2 数据流（概念）

1. **发射路径**：`sm_frontend` 在调度与记分板允许时，输出 `issue_valid`、当前 warp 的译码指令 `dec_instr`、以及从寄存器堆读出的 `issue_rs0` / `issue_rs1`（各 1024 bit SIMD 打包）。
2. **执行路径**：`sm_backend` 根据操作码将指令分派到 ALU、SFU、LSU 或 LDS；各单元通过 `valid`/`ready`（及结果 `valid_out` 等）与后端握手。
3. **写回路径**：后端产生寄存器写（`rf_wr_*`），并可能产生分支信息（`wb_branch` / `wb_branch_target`）反馈给前端更新 PC。
4. **访存路径**：`lsu` 将 load/store 翻译为对顶层的 `l1_*` 端口请求；读回应通过 `l1_resp_*` 返回给 LSU，再作为 load 数据写回路径的一部分。

### 3.3 性能与完成判定

- **`perf_inst_count`**：在 `issue_valid && issue_ready` 的周期对计数器加一，统计的是 **成功握手的发射条数**（与前端/后端流水线是否每周期一条有关，不等价于“已退役指令数”除非另行定义）。
- **`task_done`**：赋值为 `~|warp_status`，即当 `warp_status` 全为 0 时置位。`warp_status` 来自前端 `warp_active`，语义为 **各 warp 是否仍活跃**（每位对应一个 warp）。

## 4. 顶层端口说明

### 4.1 时钟与复位

| 端口 | 方向 | 说明 |
|------|------|------|
| `clk` | 输入 | 同步时钟。 |
| `rst_n` | 输入 | 异步低有效复位（与子模块一致）。 |

### 4.2 任务接口（来自 GPC）

| 端口 | 位宽 | 方向 | 说明 |
|------|------|------|------|
| `entry_pc` | 64 | 输入 | 任务入口程序计数器（起始地址）。 |
| `start_warp_id` | `WARP_ID_W` | 输入 | 起始 warp 编号，用于多 warp 任务在 SM 内的编号区间。 |
| `num_warps` | 5 | 输入 | 计划激活的 warp 数量（语义上为 GPC 侧配置）。 |
| `task_valid` | 1 | 输入 | 任务信息有效。 |
| `task_ready` | 1 | 输出 | SM 可接受该任务（与 GPC 形成 valid/ready 握手）。 |
| `task_done` | 1 | 输出 | 所有 warp 已结束；由 `~|warp_status` 得到。 |

**说明（与当前 RTL 一致）**：`num_warps` 在 `sm_core_top` 内部 **未连接到子模块**。若需按数量激活 warp，应在 `sm_frontend` 或单独控制逻辑中接入；文档保留端口语义供集成时完善。

### 4.3 L1 类存储器接口（至 GPC 路由器 / L1）

| 端口 | 位宽 | 方向 | 说明 |
|------|------|------|------|
| `l1_req_addr` | 64 | 输出 | 访存请求物理或虚拟地址（由 LSU 驱动）。 |
| `l1_req_data` | 128 | 输出 | 写数据（位宽与 LSU/L1 约定一致）。 |
| `l1_req_wr_en` | 1 | 输出 | 写使能。 |
| `l1_req_valid` | 1 | 输出 | 请求有效。 |
| `l1_req_ready` | 1 | 输入 | 下游可接受本次请求。 |
| `l1_resp_data` | 128 | 输入 | 读返回数据。 |
| `l1_resp_valid` | 1 | 输入 | 读数据有效。 |

典型用法为 **valid/ready 请求通道** 与 **响应有效** 组合，完成 load/store 的全局存储访问。

### 4.4 调试与性能

| 端口 | 位宽 | 方向 | 说明 |
|------|------|------|------|
| `perf_inst_count` | 32 | 输出 | 发射握手成功次数计数（见 3.3）。 |
| `warp_status` | `NUM_WARPS` | 输出 | 每 warp 一位：通常 1 表示活跃，0 表示空闲；全 0 时 `task_done` 为 1。 |

## 5. 内部互联要点

### 5.1 `register_file`

- **4 个读口**：连至 `sm_frontend` 的 `rf_rd_*`，供译码阶段读取源操作数。
- **2 个写口**：由 `sm_backend` 的 `rf_wr_*` 驱动；`bank_conflict` 在顶层未接（子模块仍可能输出该信号）。

### 5.2 `sm_frontend`

负责 PC、I-cache 侧接口、warp 调度、记分板以及与后端的 issue/writeback 交互。在 `sm_core_top` 中，**取指侧为占位连接**：`ifetch_data` 固定为 0、`ifetch_ready` 固定为 1，真实 I-cache 需在集成时替换。

### 5.3 `sm_backend`

将单条译码指令路由到 ALU/SFU/LSU/LDS，并处理写回与分支反馈；双写口寄存器堆接口与后端说明一致。

### 5.4 执行单元

| 实例 | 作用 |
|------|------|
| `alu_pipeline` | 向量 ALU，源/结果为 1024 bit。 |
| `sfu` | 特殊函数单元（如 rcp/rsq 等，以具体 `sfu.v` 为准）。 |
| `lsu` | 全局 load/store，连接顶层 `l1_*`；内部对部分端口使用常量或广播简化（见源码注释）。 |
| `lds_unit` | 本地数据共享；顶层对部分信号使用复制或占位，完整 LDS 数据回路与后端的衔接需结合后续集成。 |

---

## 6. 设计原理摘要

- **SIMT**：以 **warp** 为单位发射一条指令，但在数据路径上对 **32 个 lane** 并行（1024 bit 打包），与典型 GPU 执行模型一致。
- **记分板与 stall**：前端在 `~issue_ready` 或记分板 `sc_stall_raw` / `sc_stall_waw` 置位时停顿，避免 RAW/WAW 冒险。
- **多执行单元**：后端按指令类型选择 ALU/SFU/LSU/LDS，并用 `ready` 反压上游，避免单元过载。

---

## 7. 集成与验证注意点

1. **I-cache**：替换 `sm_core_top` 内对 `sm_frontend` 取指端口的占位连接，否则功能仿真无法执行真实程序流。
2. **`num_warps`**：若任务模型依赖该信号，需接入前端的激活掩码或相关启动逻辑。
3. **LSU/LDS**：当前存在为连线方便而做的广播与常量；系统级验证时应与真实内存层次、LDS 容量及 bank 规则对齐。
