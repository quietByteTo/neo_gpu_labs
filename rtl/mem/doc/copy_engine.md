这是一个 **异步 DMA 复制引擎（Copy Engine）**，用于在 **Host 内存（CPU 内存）** 和 **Device 内存（L2 缓存/HBM）** 之间高速搬运数据，是异构计算架构（如 AI 加速器/GPU）的核心数据通路组件。

---

## 1. 核心功能定位

```
┌─────────────────────────────────────────────────────────────┐
│                         Host CPU                            │
│                    (System Memory - DRAM)                   │
└─────────────┬───────────────────────────────────────────────┘
              │ PCIe x16/x8
              ▼
┌─────────────────────────────────────────────────────────────┐
│  ┌──────────┐    ┌──────────┐    ┌──────────┐            │
│  │  Copy    │◄──►│   L2     │◄──►│  HBM     │            │
│  │  Engine  │    │  Cache   │    │ Controller│            │
│  └────┬─────┘    └────┬─────┘    └────┬─────┘            │
│       │               │               │                   │
│  PCIe │          Crossbar│         HBM Channels           │
└───────┼───────────────┼───────────────────────────────────┘
        │               │
   Host Mem         Device Mem
(H2D/D2H DMA)      (GPC Compute)
```

**关键能力**：
- **双向并发**：同时支持 H2D（Host→Device）和 D2H（Device→Host）
- **描述符列表**：支持链式 DMA（最多 8 个描述符深），一次配置多段传输
- **缓存一致性**：通过 L2 Snoop 接口确保 DMA 数据与 L2 缓存同步
- **信号量同步**：与 GPC 计算单元协同（等待/通知机制）

## 3. 数据通路详解

### A. H2D（Host to Device）数据流
```
Host DRAM → PCIe Read → Copy Engine → L2 Snoop → L2 Write
```

**详细步骤**：
1. **取描述符**：从 FIFO 读取 `active_desc`（包含 Host 源地址、Device 目标地址、长度）
2. **PCIe 读请求**：`pcie_rd_addr` + `pcie_rd_len`（突发 256B）
3. **数据缓冲**：`pcie_rd_data`（512-bit）到达
4. **一致性检查**：`snoop_addr` 查询 L2，若命中可能需要无效化/写回
5. **L2 写入**：`l2_addr` + `l2_wdata` + `l2_wr_en`，握手 `l2_ready`
6. **地址更新**：`src += 256`, `dst += 256`, `remaining -= 256`

### B. D2H（Device to Host）数据流
```
L2 Read → Copy Engine → PCIe Write → Host DRAM
```

**详细步骤**：
1. **L2 读请求**：`l2_addr`（Device 源地址），`l2_wr_en=0`
2. **等待数据**：`l2_rdata_valid` 返回 512-bit 数据
3. **PCIe 写请求**：`pcie_wr_addr`（Host 目标地址）+ `pcie_wr_data`
4. **完成握手**：`pcie_wr_ready` 确认写入 PCIe 子系统
5. **循环**：直到 `bytes_remaining == 0`

## 4. 描述符格式（Descriptor）

每个描述符 64 字节（512-bit），支持 scatter-gather：

| 字段 | 位宽 | 说明 |
|------|------|------|
| `src_addr` | 64-bit | 源地址（Host PA 或 Device PA） |
| `dst_addr` | 64-bit | 目标地址（Device PA 或 Host PA） |
| `length` | 32-bit | 传输字节数（最大支持大粒度传输） |
| `control` | 32-bit | `[0]: 方向(0=H2D,1=D2H), [1]: 中断使能` |
| reserved | 320-bit | 对齐保留 |

**工作流程**：
- 驱动程序在 Host 内存构建描述符环（Ring）
- 通过 PCIe 配置寄存器写入 `desc_base_addr` 和 `desc_count`
- Copy Engine 自动遍历描述符列表，实现**链式 DMA**

## 5. 状态机与关键机制

### 主状态机
```mermaid
IDLE → FETCH_DESC → {H2D_COPY | D2H_COPY} → COMPLETE → IDLE
  ↑                                              │
  └────────────── SYNC (if semaphore) ←──────────┘
```

| 状态 | 行为 |
|------|------|
| **IDLE** | 等待 `ce_enable` 和非空描述符 FIFO |
| **FETCH_DESC** | 解析描述符，确定方向（H2D/D2H） |
| **H2D_COPY** | 突发读 PCIe → 突发写 L2，维护字节计数 |
| **D2H_COPY** | 突发读 L2 → 突发写 PCIe |
| **SYNC** | 等待计算单元信号量（`sema_in` 匹配） |
| **COMPLETE** | 更新 `bytes_copied`，发送信号量（`sema_valid`），可选中断 |

### 关键机制

#### 1. 信号量同步（与 GPC 计算协同）
```verilog
// 输入：等待特定值（数据依赖）
input  [15:0] sema_in;

// 输出：通知计算完成（生产者-消费者模型）
output [15:0] sema_out;  // 完成计数
output       sema_valid; // 脉冲信号
```
- **场景**：H2D 传输完成后通知 GPC 开始计算
- **场景**：GPC 计算完成后通过信号量触发 D2H 回拷

#### 2. L2 监听（Snoop）- 缓存一致性
```verilog
output [63:0] snoop_addr;
output        snoop_valid;
input         snoop_hit;  // L2 返回是否命中
```
- **H2D 时**：检查目标地址是否在 L2，命中则触发写回/无效化，避免Stale Data
- **一致性协议**：确保 Host 写入的数据对 Device 计算可见

#### 3. 性能计数器
- `bytes_copied`：累计传输字节数（可通过配置接口读取）
- 用于带宽监控和调试（ BW = bytes_copied / time ）

**Copy Engine 的角色**：
- **数据搬运工**：在 Host DRAM 和 Device HBM/L2 之间建立高速 DMA 通道（通常 PCIe 带宽 32-64 GB/s）
- **计算触发器**：通过信号量机制驱动 GPC 计算流水线（H2D→Compute→D2H）
- **缓存守门员**：通过 Snoop 接口维护异构内存一致性（Host 与 Device 缓存同步）

这是一个工业级 DMA 控制器设计，常见于 AI 训练芯片（如 NVIDIA GPU 的 Copy Engine、TPU 的 HBM 控制器）。
